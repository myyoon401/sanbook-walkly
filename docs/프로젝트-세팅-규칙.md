# 📐 프로젝트 세팅 규칙

> walkly 프로젝트의 코드/구조 규칙과 개인 취향을 모아두는 문서.
> 새 규칙이 생기면 여기에 바로 추가해서 단일 진실 공급원으로 관리한다.

---

## 1. 패키지 구조 — Feature + Layer 하이브리드

```
net/sanbook/walkly/
├── WalklyApplication.kt
├── {domain}/
│   ├── controller/
│   ├── service/
│   ├── repository/
│   └── entity/
└── common/
    ├── controller/
    ├── exception/
    └── ...
```

- **도메인 우선**, 그 하위에 계층 폴더를 둔다.
- 계층 폴더는 성격이 다른 파일이 한 폴더에 섞이는 걸 방지한다. ("타입이 곧 위치")
- `common/`도 동일 규칙을 따른다 (controller/, exception/ 등).
- 패키지 선언은 경로 그대로: `net.sanbook.walkly.{domain}.{layer}`.

---

## 2. 계층 분리 — Controller → Service → Repository

- **Controller에서 Repository를 직접 호출하지 않는다.** 테스트/임시 코드라도 Service를 경유.
- 이유: 테스트용이어도 예외를 두면 패턴이 무너진다. 예외 없음.

---

## 3. 도메인 간 참조 — ID로만

- 크로스 도메인에 `@ManyToOne` / `@OneToMany` 같은 JPA 연관관계를 쓰지 않는다.
- `Walk`가 `User`를 참조한다면 `val userId: UUID` (✅), `val user: User` (❌).
- 조인이 필요하면 Service에서 두 Repository를 조합하거나 JPQL/QueryDSL로 해결.
- 이유: 도메인 결합도 최소화, 순환참조 방지, 추후 모듈/서비스 분리 시 유리 (DDD Aggregate 경계).

---

## 4. API 응답/요청 — `data class` 기본

- 외부 노출되는 API의 요청/응답 타입은 **항상 `data class`로 선언**.
- `Map<String, Any>` 반환 금지 (Swagger 스키마 손실, 타입 안전성 상실).
- Request DTO와 Entity를 분리한다. 들어오는 JSON을 엔티티에 직접 매핑하면 보안 이슈(Mass Assignment) 발생.
- 예외: 헬스체크처럼 스키마가 중요하지 않은 디버그성 엔드포인트는 Map 허용.

---

## 5. ID 전략 — UUID vs Long

| 케이스 | ID 타입 |
|--------|---------|
| 외부 URL에 노출되는 엔티티 (User, Walk 등) | `UUID` |
| 내부 참조만 되고 대량 insert 일어나는 엔티티 (DailyStep, Log 등) | `Long` |

- UUID는 enumeration 공격 방어 + 분산 환경 친화적.
- Long은 B-Tree 인덱스 성능·저장공간 유리.
- 성능이 걱정되면 **UUID v7**(시간 정렬) 고려.

---

## 6. 타임스탬프 — `@CreationTimestamp` / `@UpdateTimestamp` 또는 Auditing

- `createdAt`, `updatedAt`은 Hibernate 어노테이션(`@CreationTimestamp`, `@UpdateTimestamp`) 또는 Spring Data Auditing(`@CreatedDate`, `@LastModifiedDate`)으로 자동 관리.
- 앱 레벨에서 `LocalDateTime.now()`를 기본값으로 쓰는 것도 허용하지만, 서버 다중화나 배치 처리 있을 때는 DB/Hibernate 레벨로 이동.
