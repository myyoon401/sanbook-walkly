# 🎯 미션 1. Walker → User 리팩토링

**목표:** `Walker` 엔티티를 `User`로 재설계하면서 Kotlin의 `data class` / nullable / `LocalDateTime`을 경험한다.

---

## 📝 요구사항

### 1. 엔티티 이름 변경 & 필드 추가

`entity/Walker.kt` → `entity/User.kt`

| 필드명 | 타입 | 제약 |
|--------|------|------|
| `id` | `UUID` | PK, 기본값 `UUID.randomUUID()` |
| `email` | `String` | NOT NULL, UNIQUE |
| `nickname` | `String` | NOT NULL |
| `profileImageUrl` | `String?` | nullable (선택) |
| `createdAt` | `LocalDateTime` | NOT NULL, 기본값 `LocalDateTime.now()` |

> 💡 `stepCount`는 User에서 제거합니다. (다음 미션에서 `DailyStep`으로 분리할 예정)
>
> 💡 **왜 UUID?** 사용자 ID는 URL(`/users/{id}`)에 노출되므로, 순차 ID를 쓰면 전체 가입자 수 유추·enumeration 공격이 가능합니다. 단, 대량 insert가 일어나는 내부 엔티티(예: `DailyStep`)는 `Long`이 인덱스 성능상 유리하니 테이블마다 판단하세요.
>
> 💡 성능이 걱정되면 **UUID v7**(시간 정렬 가능)을 고려하세요. 이번 미션에서는 학습 목적상 `UUID.randomUUID()`(v4)로 충분합니다.

### 2. Repository / Service / Controller 동반 수정

- `WalkerRepository` → `UserRepository`
- `WalkerService` → `UserService`
- `DbTestController`의 엔드포인트는 일단 유지하되 User 조회로 바꿔도 되고, 삭제해도 됨 (본인 판단)

### 3. API 1개만 만들기

```
POST /api/users
Request Body: { "email": "...", "nickname": "...", "profileImageUrl": "..." }
Response: 생성된 User
```

이메일 중복 시 에러 응답 (HTTP 409 Conflict).

## 🔍 이번 미션의 Kotlin 학습 포인트

1. **`?` 붙은 nullable 타입** — `String?`과 `String`의 차이 체감하기
2. **기본값 파라미터** — `createdAt: LocalDateTime = LocalDateTime.now()`
3. **`data class` vs `class`** — Request DTO는 `data class`로 만들어보기 (equals/hashCode 자동생성)
4. **확장함수 맛보기** — 선택: `User.toResponse()` 같은 확장함수로 엔티티→DTO 변환해보기

## 💡 힌트 (막히면 펼쳐보세요)

- Request DTO와 Entity를 분리하세요. 외부에서 들어오는 JSON을 그대로 엔티티에 매핑하면 보안 이슈 생깁니다.
- 이메일 중복 체크: `userRepository.existsByEmail(email)` — Spring Data JPA는 메서드 이름만으로 쿼리를 만들어줍니다.
- 409 응답: `ResponseStatusException(HttpStatus.CONFLICT, "이미 존재하는 이메일입니다")`
- UUID 컬럼 매핑 (PostgreSQL 기준):
  ```kotlin
  @Id
  @Column(columnDefinition = "uuid")
  val id: UUID = UUID.randomUUID()
  ```
