# 🎯 미션 2. 일일 걸음수 기록 API

**목표:** `DailyStep` 엔티티를 추가해 "하루 하나의 걸음수 기록"을 저장한다. 이 과정에서 **크로스 도메인 ID 참조(규칙 3)**, **Long ID 전략(규칙 5)**, **Kotlin의 `LocalDate`와 nullable 반환**을 실전으로 경험한다.

> 💡 인증은 아직 붙이지 않는다. `accountId`는 요청 body로 받아 처리한다.
> 추후 JWT가 붙으면 token에서 뽑아 쓰도록 교체할 예정.

---

## 📝 요구사항

### 1. `DailyStep` 엔티티 생성

위치: `net.sanbook.walkly.dailystep.entity.DailyStep`

| 필드명 | 타입 | 제약 |
|--------|------|------|
| `id` | `Long` | PK, `@GeneratedValue(strategy = GenerationType.IDENTITY)` |
| `accountId` | `UUID` | NOT NULL |
| `date` | `LocalDate` | NOT NULL |
| `stepCount` | `Int` | NOT NULL |
| `createdAt` | `LocalDateTime` | NOT NULL, `@CreationTimestamp` |
| `updatedAt` | `LocalDateTime` | NOT NULL, `@UpdateTimestamp` |

추가 제약:
- `(accountId, date)` **복합 UNIQUE** — 한 사용자당 하루 한 레코드
- `@Table(uniqueConstraints = [UniqueConstraint(columnNames = ["account_id", "date"])])` 로 지정

> 💡 **왜 Long?** `DailyStep`은 유저 수 × 날짜 수만큼 대량 insert가 일어나는 내부 엔티티다. URL에 노출되지 않으므로 UUID 대신 Long이 인덱스/저장공간상 유리 (규칙 5).
>
> 💡 **왜 `val account: Account`가 아니라 `val accountId: UUID`?** 규칙 3 — 도메인 간 참조는 ID로만 한다. 조인이 필요해지면 Service에서 두 Repository를 조합한다.
>
> 💡 **`LocalDate` vs `LocalDateTime`?** 하루 단위 기록이라 시분초 정보가 필요 없음. 타입을 좁히면 의미도 좁혀진다.

### 3. API 1개: 걸음수 Upsert

```
POST /daily-steps
Request Body:
{
  "accountId": "b0f5c0d0-...",
  "date": "2026-04-20",
  "stepCount": 8423
}
Response: 저장된 DailyStep
```

**동작 방식 (Upsert):**
- 해당 `(accountId, date)` 레코드가 **없으면** 새로 생성
- **있으면** `stepCount`만 덮어쓰고 `updatedAt` 갱신
- 응답은 항상 200 OK + 최종 상태의 리소스

> 💡 Upsert 로직은 Service에서 `findByAccountIdAndDate()`로 조회 → 있으면 update, 없으면 new. DB 레벨 UPSERT(ON CONFLICT 등)는 이번엔 사용하지 않는다 (학습 흐름 우선).

### 4. 검증

- `stepCount`가 **음수면** 400 Bad Request
  - 힌트: `ResponseStatusException(HttpStatus.BAD_REQUEST, "...")`
- `accountId`가 **존재하지 않는 계정이면** 404 Not Found
  - 힌트: `accountRepository.existsById(accountId)` 로 체크

> 💡 규칙 3에 따라 JPA 연관관계는 안 걸지만, 유효성 검증은 Service에서 명시적으로 해준다. 이게 "느슨한 참조"의 대가.

## 🔍 이번 미션의 Kotlin 학습 포인트

### 1. `findByAccountIdAndDate()` 반환 타입은 `DailyStep?`
Java의 `Optional<DailyStep>` 대신 **nullable**로 표현한다. 체이닝 시 쓸 수 있는 연산자를 익혀보자:

```kotlin
val existing: DailyStep? = repo.findByAccountIdAndDate(accountId, date)

// 패턴 A: if-let 스타일
if (existing != null) {
    existing.stepCount = request.stepCount  // update
    repo.save(existing)
} else {
    repo.save(request.toEntity())  // insert
}

// 패턴 B: ?.let { } / ?: (엘비스)
existing?.apply { stepCount = request.stepCount }
    ?: request.toEntity()
```

두 패턴 다 써보고 가독성을 비교해보자.

### 2. Upsert하려면 `stepCount`는 `var`여야 한다
지금까지는 엔티티 필드를 전부 `val` (immutable)로 만들었다. update가 필요한 필드는 `var`로 선언한다.

> 💡 Kotlin의 `val`은 Java의 `final`이고, `var`은 일반 변수다. JPA 엔티티에서 mutable 필드를 명시적으로 구분해주는 효과가 있다.

### 3. `LocalDate` 파싱
Jackson은 ISO 포맷(`"2026-04-20"`)을 `LocalDate`로 자동 변환해준다. 별도 설정 불필요 (`jackson-module-kotlin` + `jsr310`은 이미 Spring Boot 기본 제공).

### 4. 복합 UNIQUE 제약
```kotlin
@Entity
@Table(
    name = "daily_step",
    uniqueConstraints = [
        UniqueConstraint(columnNames = ["account_id", "date"])
    ]
)
class DailyStep(...)
```

> 💡 Kotlin에서 어노테이션 배열 인자는 `[ ]` (arrayOf 대신).

### 5. @GeneratedValue IDENTITY
UUID는 앱 레벨에서 `UUID.randomUUID()` 할당했지만, Long PK는 DB auto-increment에 맡긴다:
```kotlin
@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
val id: Long = 0L
```
save() 전에는 `0L`, save() 후 Hibernate가 실제 값으로 덮어쓴다.

## 💡 힌트

- Repository 메서드: `findByAccountIdAndDate(accountId: UUID, date: LocalDate): DailyStep?`
- Account 존재 체크는 `AccountRepository.existsById(accountId)` — 별도 Service를 거칠지 Repository 직접 부를지는 판단 필요 (규칙 2 상기: Controller에서 Repository 직접 호출 금지. **Service에서 다른 도메인 Repository 호출은 허용**)

  ```bash

    -H "Content-Type: application/json" \
  ```

## 🤔 고민해볼 것 (선택)

- `POST /daily-steps`가 생성/수정 둘 다 처리하는 건 REST 관점에선 어색하다. `PUT /accounts/{id}/daily-steps/{date}`가 더 RESTful할지도? 둘 다 장단점 있으니 본인 취향대로.
- `stepCount`를 누적(+=)으로 할지 덮어쓰기(=)로 할지 — 스마트폰에서 동기화 시나리오를 생각하면 덮어쓰기가 보통 맞음 (기기가 이미 누적값을 보내니까).
