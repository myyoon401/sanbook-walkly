# 🎯 미션 4. 걸음수 랭킹 API (Top N)

**목표:** 여러 계정의 걸음수를 집계해 **기간별 랭킹**을 뽑는 API를 만든다. 이 과정에서 Kotlin의 **`when` 표현식**, **`groupBy` + `mapValues` + `sumOf` 체이닝**, **`enum class` with method**, **`mapIndexed`**, **크로스 도메인 조회(규칙 3 실전)** 를 익힌다.

> 💡 인증은 여전히 붙이지 않는다. 랭킹은 공개 API로 본다.

---

## 📝 요구사항

### 1. API 1개: 기간별 Top N 랭킹

```
GET /daily-steps/rankings?period=WEEK&limit=10
```

**쿼리 파라미터:**
- `period`: `TODAY` | `WEEK` | `MONTH` (기본값 `TODAY`)
  - `TODAY`: 오늘 하루
  - `WEEK`: 오늘 포함 최근 7일
  - `MONTH`: 오늘 포함 최근 30일
- `limit`: `1..100` (기본값 `10`)

**응답 (`DailyStepRankingResponse`):**
```json
{
  "period": "WEEK",
  "from": "2026-04-18",
  "to": "2026-04-24",
  "rankings": [
    { "rank": 1, "accountId": "b0f5c0d0-...", "nickname": "홍길동", "totalSteps": 85230 },
    { "rank": 2, "accountId": "a3e1d2f4-...", "nickname": "김영희", "totalSteps": 78412 },
    { "rank": 3, "accountId": "c7b2a8e9-...", "nickname": "이철수", "totalSteps": 64500 }
  ]
}
```

- `totalSteps` = 해당 기간 내 그 계정의 총 걸음수
- 기록이 전혀 없는 계정은 응답에서 **제외**
- `rank`는 1부터 시작
- 동점자는 이번엔 단순하게 **나오는 순서**로 처리 (stable sort). tie-break는 선택 과제.

### 2. 검증
- `limit`이 1 미만 또는 100 초과 → **400 Bad Request**
- `period`는 enum 매핑 실패 시 Spring이 자동으로 400 (별도 처리 불필요)

## 🔍 이번 미션의 Kotlin 학습 포인트

### 1. `enum class` with method (+ `when` 표현식) 🆕

랭킹 기간을 **enum**으로 표현하되, 각 값에 계산 로직을 덧붙인다.

```kotlin
enum class RankingPeriod {
    TODAY, WEEK, MONTH;

    fun toDateRange(today: LocalDate = LocalDate.now()): Pair<LocalDate, LocalDate> =
        when (this) {
            TODAY -> today to today
            WEEK  -> today.minusDays(6) to today
            MONTH -> today.minusDays(29) to today
        }
}
```

**배울 점:**
- `when`은 **표현식** — 값을 반환한다 (자바 `switch/case`가 아님)
- `when`에 enum을 주면 **exhaustive** — 누락된 case가 있으면 컴파일러가 경고/에러
- enum에 메서드 붙이기가 자바보다 훨씬 자연스러움
- `to` 중위 함수로 `Pair` 생성 — `"a" to 1` == `Pair("a", 1)`

### 2. `groupBy` + `mapValues` + `sumOf` 체이닝 🆕♻️

```kotlin
val totalsByAccount: Map<UUID, Int> = steps
    .groupBy { it.accountId }                             // Map<UUID, List<DailyStep>>
    .mapValues { (_, list) -> list.sumOf { it.stepCount } } // Map<UUID, Int>
```

**배울 점:**
- `groupBy { keySelector }` → 그룹 key로 묶은 `Map<K, List<T>>`
- `mapValues { (k, v) -> ... }` → key는 그대로, value만 변환
- `sumOf { }` — 미션 3에서 썼던 걸 **그룹 안에서 다시 쓰기** (반복 훈련)
- 구조 분해 `(_, list)` — Pair/Map.Entry를 바로 풀 수 있음

### 3. `sortedByDescending` + `take(n)` + `mapIndexed` 🆕

```kotlin
val rankings = totalsByAccount.entries
    .sortedByDescending { it.value }
    .take(limit)
    .mapIndexed { idx, (accountId, total) ->
        RankingEntry(
            rank = idx + 1,
            accountId = accountId,
            nickname = nicknames[accountId] ?: "(탈퇴)",
            totalSteps = total,
        )
    }
```

**배울 점:**
- `mapIndexed` — **인덱스가 필요할 때** `map` 대신 쓴다. "순위 매기기"의 정석.
- `take(n)` = 상위 n개, `drop(n)` = 앞 n개 버림
- `sortedByDescending` vs `sortedBy` — 둘 다 새 리스트 반환 (원본 불변)

### 4. 크로스 도메인 조회 — `accountId` → `nickname` 매핑 🆕

규칙 3에 따라 `DailyStep.account: Account` 같은 JPA 조인은 없다. Service에서 **두 도메인을 조합**한다.

```kotlin
// 1. 집계 대상 accountId 목록
val accountIds: Set<UUID> = totalsByAccount.keys

// 2. Account 일괄 조회 → nickname map
val nicknames: Map<UUID, String> = accountService.findAllByIds(accountIds)
    .associate { it.id to it.nickname }
```

**배울 점:**
- `associate { }` → 각 요소에서 `Pair<K, V>`를 만들어 `Map<K, V>` 생성
- `associateBy { }` → 요소 자체를 value로, key만 추출 — `associateBy { it.id }` 는 `Map<UUID, Account>`
- **N+1 금지** — `findAllByIdIn(ids)` 한 번의 쿼리로 모든 Account 조회

> 💡 `AccountRepository`에 `findAllByIdIn(ids: Collection<UUID>): List<Account>` 추가. Spring Data JPA의 `In` 키워드 활용.

### 5. 엘비스 `?:` 재등장 ♻️

```kotlin
nickname = nicknames[accountId] ?: "(탈퇴)"
```

Map 접근은 항상 nullable (`V?` 반환). 폴백 필수. **미션 3에서 배운 패턴을 또 한 번.**

### 6. `require` 검증 + Range 문법 🆕

```kotlin
require(limit in 1..100) { "limit은 1~100 범위여야 합니다" }
```

- `require { }` 자체는 복습
- **`in 1..100`** Range 문법이 새로움. `1..100` = `IntRange`, `in`은 포함 여부 체크
- `in` 연산자는 내부적으로 `contains()` 호출

### 7. `Pair` & 구조 분해 🆕

`Pair` 는 Kotlin 기본 제공 튜플. `to` 중위함수로 만들고, 받을 땐 구조 분해.

```kotlin
val (from, to) = period.toDateRange()  // Pair<LocalDate, LocalDate> 분해

totalsByAccount.entries.mapIndexed { idx, (id, total) ->  // Map.Entry 분해
    ...
}
```

> 💡 반환값이 2개뿐일 때 `Pair` 는 편하지만, 의미가 중요하면 **전용 data class**가 낫다. 3개 이상은 고민 없이 data class.

### Repository 추가
```kotlin
// DailyStepRepository

// AccountRepository
fun findAllByIdIn(ids: Collection<UUID>): List<Account>
```

### AccountService에 메서드 노출
Controller → AccountService → AccountRepository 레이어 유지. DailyStepService가 AccountRepository를 직접 부르지 않도록:

```kotlin
// AccountService

    accountRepository.findAllByIdIn(ids)
```

### enum 쿼리 파라미터 바인딩
Spring은 `@RequestParam` 타입이 enum이면 문자열 → enum 자동 매핑.
```kotlin
@RequestParam(defaultValue = "TODAY") period: RankingPeriod
```
대소문자 정확히 맞춰야 함 (`week` 는 400). 대소문자 무시하려면 custom `Converter` 등록 — **선택**.

### Service 전체 흐름 스케치
```
1. require(limit in 1..100)
2. val (from, to) = period.toDateRange()
3. val steps = dailyStepRepository.findAllByDateBetween(from, to)
4. val totalsByAccount = steps.groupBy { it.accountId }
                              .mapValues { (_, list) -> list.sumOf { it.stepCount } }
5. val nicknames = accountService.findAllByIds(totalsByAccount.keys)
                                 .associate { it.id to it.nickname }
6. val rankings = totalsByAccount.entries
                                 .sortedByDescending { it.value }
                                 .take(limit)
                                 .mapIndexed { idx, (id, total) -> RankingEntry(...) }
7. return DailyStepRankingResponse(period, from, to, rankings)
```

### Response DTO (중첩 data class 권장)
```kotlin
data class DailyStepRankingResponse(
    val period: RankingPeriod,
    val from: LocalDate,
    val to: LocalDate,
    val rankings: List<RankingEntry>,
) {
    data class RankingEntry(
        val rank: Int,
        val accountId: UUID,
        val nickname: String,
        val totalSteps: Int,
    )
}
```

## 🤔 고민해볼 것 (선택)

1. **DB 집계(`@Query` + GROUP BY) vs 애플리케이션 집계(`groupBy`)** — 이번엔 학습 목적상 애플리케이션 집계. 실서비스에서 수만 명 랭킹이면 DB 집계가 필수. 언제 선을 넘어야 할지 기준 정리해보기.

2. **동점자 처리** — `[1, 2, 2, 4]` (Standard competition rank) vs `[1, 2, 2, 3]` (Dense rank) vs `[1, 2, 3, 4]` (순서대로). 어느 쪽이 UX에 자연스러운지?

3. **캐시?** — "오늘 랭킹"은 1분~1시간에 한 번만 재계산해도 충분할 수 있다. `@Cacheable` 어디에 붙일지, 무효화 타이밍은?

4. **N+1 문제** — 지금은 쿼리 2번 (DailyStep + Account)으로 끝남. 만약 JPA 연관관계(`@ManyToOne Account`)를 걸었다면 LAZY 로딩 중 N+1 폭발. **규칙 3의 장점을 체감해볼 것.**

5. **닉네임 탈퇴/누락** — `nicknames[accountId]` 가 null인 경우는 언제? (힌트: DailyStep 기록 후 Account 탈퇴). `"(탈퇴)"` 로 보여줘도 되는지, 아예 제외할지?

6. **`Pair` vs 전용 data class** — `toDateRange()` 반환을 `Pair<LocalDate, LocalDate>` 로 했다. `DateRange(from, to)` data class가 더 나았을까? 어느 시점에 Pair에서 data class로 승격하는 게 맞을까?
