# 🎯 미션 3. 걸음수 조회 API (단건 + 기간 요약)

**목표:** 이제 저장한 걸음수를 **읽어오는** API를 만든다. 이 과정에서 Kotlin의 **컬렉션 함수(`sumOf`/`maxByOrNull`/`map`)**, **스코프 함수(`let`/`run`)**, **확장 프로퍼티**, **기본 파라미터값**, **엘비스 연산자 심화 활용**을 실전으로 익힌다.

> 💡 여전히 인증은 붙이지 않는다. `accountId`는 path variable로 받아 처리한다. (JWT 도입 시 token에서 주입으로 교체)

---

## 📝 요구사항

### 1. API 2개 추가

#### API-1. 단건 조회 — 특정 날짜의 걸음수

```
GET /accounts/{accountId}/daily-steps?date=2026-04-20
```

**동작:**
- `date` 쿼리 파라미터 **생략 가능**, 생략 시 **오늘**
- `(accountId, date)` 레코드가 **없으면 404**
- 있으면 `DailyStepResponse` (미션 2에서 쓰던 것 재사용)

> 💡 `date` 생략 시 "오늘"로 처리 — 이게 **기본 파라미터값**이 빛나는 지점. Kotlin에서는 컨트롤러 핸들러 시그니처에 `= LocalDate.now()` 형태로 녹일 수도 있고, `@RequestParam(required = false)` + 서비스에서 `?: LocalDate.now()`로 처리할 수도 있다. **두 방식 다 써보고 각자의 장단점을 정리해본다.**

#### API-2. 기간 요약 — 걸음수 통계

```
GET /accounts/{accountId}/daily-steps/summary?from=2026-04-01&to=2026-04-20
```

**응답 (`DailyStepSummaryResponse`):**
```json
{
  "accountId": "b0f5c0d0-...",
  "from": "2026-04-01",
  "to": "2026-04-20",
  "daysRecorded": 15,
  "totalSteps": 123456,
  "averageSteps": 8230,
  "maxDay": { "date": "2026-04-18", "stepCount": 15230 }
}
```

- `daysRecorded` = 실제로 기록된 날 수 (기록 없는 날은 0으로 채우지 않고 **없다고 본다**)
- `averageSteps` = `totalSteps / daysRecorded` (정수 나눗셈, `daysRecorded == 0`이면 0)
- `maxDay` = 그 기간 중 가장 많이 걸은 날. 기록 없으면 `null`.

### 2. 검증

- `from > to` → **400 Bad Request** ("from은 to보다 늦을 수 없습니다")
- 기간이 **31일 초과**(`ChronoUnit.DAYS.between(from, to) > 31`) → **400 Bad Request** ("최대 31일까지 조회 가능")
- `accountId` 존재하지 않음 → **404 Not Found** (미션 2의 `validateAccountExists` 재사용)

> 💡 31일 제한은 학습용 가드. 실서비스에선 더 긴 기간 조회 시 페이징/집계 테이블을 따로 설계하지만, 이번엔 "전제조건을 Service 진입부에서 명시적으로 던지는 감각"만 잡자. `require { ... }`를 써도 되고 `throw ResponseStatusException(...)`을 써도 된다.

## 🔍 이번 미션의 Kotlin 학습 포인트

### 1. 기본 파라미터값 (Default Arguments)

Kotlin 함수는 매개변수에 기본값을 줄 수 있다. 자바 오버로딩 대신 이걸로 끝남.

```kotlin
fun getDailyStep(accountId: UUID, date: LocalDate = LocalDate.now()): DailyStepResponse {
    // ...
}

// 호출 시:
getDailyStep(accountId)                    // date는 오늘로 채워짐
getDailyStep(accountId, LocalDate.parse("2026-04-20"))
```

Spring MVC 컨트롤러에서는 다음처럼 조합된다:

```kotlin
@GetMapping("/accounts/{accountId}/daily-steps")
fun getDailyStep(
    @PathVariable accountId: UUID,
    @RequestParam(required = false) date: LocalDate?
): DailyStepResponse = dailyStepService.getDailyStep(accountId, date ?: LocalDate.now())
```

**두 가지 방식 비교:**
- **A. 컨트롤러에서 `?: LocalDate.now()`로 폴백**
- **B. 서비스 함수 시그니처에 `date: LocalDate = LocalDate.now()` 기본값**

B는 "서비스 단독으로도 기본값이 살아있다"는 장점. A는 "기본값 계산(now())이 컨트롤러에서 명시적으로 보임"이라는 장점. 본인 취향으로 선택하고 **이유**를 회고에 남길 것.

### 2. 엘비스 연산자 `?:` 심화

미션 2에서 `?: request.toEntity()`로 **대체값 제공**에 썼다. 이번엔 같은 `?:`의 다른 얼굴 — **throw로 연결**:

```kotlin
val dailyStep = dailyStepRepository.findByAccountIdAndDate(accountId, date)
    ?: throw ResponseStatusException(HttpStatus.NOT_FOUND, "기록이 없습니다")
```

Kotlin에서 `throw`도 **표현식**이다. 그래서 `?:` 오른쪽에 올 수 있다. 자바의 `Optional.orElseThrow()`와 같은 역할인데 문법이 훨씬 짧다.

### 3. 컬렉션 함수 — `sumOf`, `maxByOrNull`, `map`, `filter`

요약 계산은 **명령형 루프 금지**, Kotlin 컬렉션 함수로만 쓴다.

```kotlin
val steps: List<DailyStep> = repo.findAllByAccountIdAndDateBetween(accountId, from, to)

val totalSteps: Int = steps.sumOf { it.stepCount }
val maxDay: DailyStep? = steps.maxByOrNull { it.stepCount }
val averageSteps: Int = if (steps.isEmpty()) 0 else totalSteps / steps.size
```

`maxByOrNull` vs `maxBy` — 후자는 **빈 리스트면 예외**. Kotlin은 nullable을 명시적으로 취급하는 쪽을 선호. 둘 다 존재하는 이유와 선택 기준을 익혀두자.

### 4. 확장 프로퍼티 (Extension Property)

확장 **함수**는 미션 1에서 `toEntity()`, `toResponse()`로 썼다. **프로퍼티**도 가능하다.

```kotlin
val List<DailyStep>.totalSteps: Int
    get() = sumOf { it.stepCount }

val List<DailyStep>.maxDay: DailyStep?
    get() = maxByOrNull { it.stepCount }

// 사용:
val summary = steps.totalSteps   // steps.sumOf { ... } 대신 프로퍼티처럼
```

**언제 쓰나?** "계산이 간단하고, 인자 없이, 의미가 명사(상태)로 읽힐 때." `steps.totalSteps`는 "steps의 총 걸음수"로 읽히니까 자연스럽다. 반면 계산이 무겁거나 인자가 필요하면 함수로.

### 5. 스코프 함수 — `let` vs `run` vs `also` vs `apply`

| 함수 | 블록 안 this/it | 반환 | 전형적 쓰임 |
|------|-----------------|------|-------------|
| `let { it -> ... }` | `it` | 블록 결과 | nullable 처리, 변환 |
| `run { this -> ... }` | `this` | 블록 결과 | 객체로 뭔가 계산 |
| `also { it -> ... }` | `it` | 객체 자신 | 로깅/부수효과 |
| `apply { this -> ... }` | `this` | 객체 자신 | 초기화/설정 (미션 2에서 사용) |

이번 미션에서 `let`을 써볼 만한 곳:

```kotlin
// 패턴 1 — nullable + 변환
val maxDayResponse: MaxDayResponse? = steps.maxByOrNull { it.stepCount }
    ?.let { MaxDayResponse(it.date, it.stepCount) }
// null이면 null, 값 있으면 변환된 DTO
```

```kotlin
// 패턴 2 — run으로 여러 값 모아 계산
val summary = steps.run {
    DailyStepSummaryResponse(
        accountId = accountId,
        from = from,
        to = to,
        daysRecorded = size,
        totalSteps = sumOf { it.stepCount },
        averageSteps = if (isEmpty()) 0 else sumOf { it.stepCount } / size,
        maxDay = maxByOrNull { it.stepCount }?.let { MaxDayResponse(it.date, it.stepCount) }
    )
}
```

**⚠️ 주의:** 스코프 함수는 남발하면 **읽기 어려워진다**. "이 줄에서 나는 뭘 반환하려는 거지?" 가 한눈에 안 보이면 일반 변수 대입이 낫다. 미션 2에서 `?.apply { }`가 빛났던 건 "객체 자체를 쓸 거라는 의도"가 명확해서였다. 두 패턴(스코프 함수 vs 일반 변수) 다 써보고 **각자의 가독성 비교**를 회고에 남길 것.

### 6. `require { }` — 전제조건 검증

Kotlin은 전제조건 검증용 빌트인 함수를 제공한다.

```kotlin
fun getSummary(accountId: UUID, from: LocalDate, to: LocalDate): DailyStepSummaryResponse {
    require(!from.isAfter(to)) { "from은 to보다 늦을 수 없습니다" }
    require(ChronoUnit.DAYS.between(from, to) <= 31) { "최대 31일까지 조회 가능" }
    // ...
}
```

`require`는 조건이 false면 `IllegalArgumentException`을 던진다. Spring에선 `@ControllerAdvice`로 이걸 400으로 매핑할 수 있지만, 아직 글로벌 예외 처리기 안 만들었으니 이번엔 `ResponseStatusException`을 직접 던져도 OK. **둘 다 써보고 차이 체감**.

### 7. `LocalDate` 계산

```kotlin
import java.time.temporal.ChronoUnit

val days = ChronoUnit.DAYS.between(from, to)   // Long, 음수 가능
from.isAfter(to)                                // Boolean
from.isBefore(to)
from.plusDays(1)
from == to                                      // equals로 비교됨 (내부에서 equals 정의됨)
```

자바와 같지만 Kotlin에서는 `==`가 자바의 `equals()`라는 점만 기억.

## 💡 힌트

- Repository 메서드: `findAllByAccountIdAndDateBetween(accountId: UUID, from: LocalDate, to: LocalDate): List<DailyStep>`
  - `Between`은 Spring Data JPA 키워드. 양끝 포함(inclusive) 쿼리가 만들어진다.
- 단건 조회는 미션 2의 `findByAccountIdAndDate`를 그대로 재사용.
- `DailyStepSummaryResponse` 내부 `maxDay`는 nested `data class`로:

  ```kotlin
  data class DailyStepSummaryResponse(
      val accountId: UUID,
      val from: LocalDate,
      val to: LocalDate,
      val daysRecorded: Int,
      val totalSteps: Int,
      val averageSteps: Int,
      val maxDay: MaxDay?
  ) {
      data class MaxDay(val date: LocalDate, val stepCount: Int)
  }
  ```

  `DailyStepSummaryResponse.MaxDay`처럼 스코프가 내부로 들어가서 이름 충돌 걱정이 없고, 쓰임이 이 응답에 한정된다는 게 명확해진다.

- URL에 `{accountId}/daily-steps`를 두는 건 **"DailyStep은 Account의 서브리소스"** 라는 REST 관점. 이게 **규칙 3(도메인 간 참조는 ID로만)** 과 충돌하지 않는다는 점 음미해볼 것 — 규칙 3은 JPA 엔티티 관계 얘기, URL 구조는 별개.

## 🤔 고민해볼 것 (선택)

1. **`date` 기본값을 컨트롤러 vs 서비스 어디서 채울지** — 서비스에서 기본값이 살아있으면 스케줄러나 다른 서비스가 호출해도 일관됨. 반면 컨트롤러에서 채우면 "HTTP 레이어의 관습"임이 명시됨. 본인 기준 만들기.

2. **없는 날짜 응답** — 404가 맞나? `stepCount: 0`으로 돌려주는 게 나은가? 클라이언트 입장에서 생각해보자. "달력 UI에 0걸음으로 표시"하려면 어느 쪽이 편한가?

3. **기간 제한 31일은 누구의 책임?** — 클라이언트(UI)에서 강제할 수도, 서버에서 강제할 수도 있다. **두 군데 다 해야 하는 이유**를 생각해보자 (힌트: 클라이언트는 믿을 수 없다).

4. **`maxByOrNull` vs `maxBy` 삭제 이력** — `maxBy`는 Kotlin 1.4에서 deprecate됐고 1.7에 제거됐다. 왜 굳이 제거까지 했을까? (힌트: Kotlin의 null 안정성 철학)
